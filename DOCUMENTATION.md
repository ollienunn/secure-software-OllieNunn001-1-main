## Overview of Key Files & Folders

### `app.py`
- **Main entry point** of the Flask application.
- Responsible for initializing the server and registering blueprints and middleware.

### `requirements.txt`
- Python dependency list used by `pip3 install -r requirements.txt`.

---

## `database/`

Contains JSON-based storage and a helper module for read/write operations.

```
database/
в”њв”Ђв”Ђ __init__.py
в”њв”Ђв”Ђ data.py
в”њв”Ђв”Ђ users.database.json
в”њв”Ђв”Ђ games.database.json
в””в”Ђв”Ђ sessions.database.json
```

- `data.py`: Contains helper functions to load/save data from the above JSON files.
- `*.json`: Used as the **persistent flat file storage** for users, games, and sessions.

---

## `core/`

Contains core logic grouped by feature. These are the **raw logic handlers**, separate from routing.

```
core/
в”њв”Ђв”Ђ auth_core.py
в”њв”Ђв”Ђ game_core.py
в”њв”Ђв”Ђ game_helpers.py
в””в”Ђв”Ђ player_core.py
```

### Files:
- `auth_core.py`: Auth logic (register, login, logout, token/CSRF management).
- `game_core.py`: Game lifecycle logic (create, update, start, end).
- `game_helpers.py`: Utility helpers shared across game modules.
- `player_core.py`: Handles logic related to player interaction (joining, answering, score tracking).

---

## `services/`

Flask **route handlers (controllers)** that act as the API interface.

```
services/
в”њв”Ђв”Ђ auth.py
в”њв”Ђв”Ђ game.py
в””в”Ђв”Ђ player.py
```

### Files:
- `auth.py`: Defines `/auth/...` endpoints, uses `auth_core.py`.
- `game.py`: Defines `/game/...` endpoints, calls into `game_core.py`.
- `player.py`: Defines `/player/...` endpoints, uses `player_core.py`.

---

## `classes/`
Contains **data structures or helper classes** used throughout the backend logic.

- `User.py`:  
  Defines the `User` class that handles most authentication-related logic and session data.

- `Error.py`:
  Defines the custom exception types used in route handling:
  - `InputError`: Raised for invalid inputs or bad requests (maps to HTTP 400).
  - `AccessError`: Raised when a user is not authorised to access a resource (maps to HTTP 403).

---

## `utils/`

General-purpose helpers used across modules.
- Examples: token generation, hashing, timestamp utils, etc.

---

## `tests/`

Contains automated **unit/integration tests** for the backend.

- Run tests with:
```bash
pytest
```
> [!WARNING]  
> Ensure you run `pytest` from the **root directory**.  
> Otherwise, PyTest wonвЂ™t recognise your imports and dependencies correctly.

## Virtual Environment (`venv/`)

- Python virtual environment for dependency isolation.
- Can be regenerated using:
```bash
python3 -m venv venv
source venv/bin/activate
pip3 install -r requirements.txt
```

> вљ пёЏ Do **not** commit `venv/` to GitHub. It should be in `.gitignore`.

## File Overview

### `backend.config.json`
Stores configuration for connecting to the backend API (e.g. base URL, endpoints).

### `cypress/` & `cypress.config.js`
Contains **end-to-end (E2E)** tests using Cypress, and its configuration file.

### `dist/`
Build output generated by Vite.  
> вљ пёЏ This folder is auto-generated and should **not** be committed to Git.

### `eslint.config.js`
Configuration for ESLint, enforcing code style and linting rules.

### `index.html`
The entry HTML file for the app. Vite injects scripts here when serving/building.

### `jsconfig.json`
Configures module resolution for VS Code, enabling absolute imports from `src/`.

### `node_modules/`
All installed dependencies from `npm install`.  
> вљ пёЏ Do not commit this folder to Git вЂ” itвЂ™s in `.gitignore`.

### `package.json` / `package-lock.json`
Dependency list, scripts, and metadata for the frontend project.

### `public/`
Static assets that will be copied to the root of the build output without processing.

### `upload/`
Folder for handling uploaded media/assets in the frontend.

### `vite.config.js`
Configuration for the Vite bundler, including plugins and server settings.
